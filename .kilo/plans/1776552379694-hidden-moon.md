# Permission-Based Access Control System

## 1. Architecture Overview

### Where Permissions Live
- **PermissionsContext**: A React context that stores and provides access to user permissions
- **Client-side storage**: Persist permissions in sessionStorage alongside auth tokens (key: `user_permissions`)
- **API endpoint**: Uses existing `usersApi.getCurrentProfile()` which returns `CurrentUserProfileDto` with `permissions: string[]` and `hasAllPermissions: boolean`
- **Flow**: API response (`/api/v1/users/profile`) → PermissionsContext → Components/Hooks/Routes

> **Note**: Backend already returns permissions in `/users/profile` response. The frontend just needs to capture and use them.

### Permission Data Flow
```
/users/profile API 
    ↓
permissions: string[], hasAllPermissions: boolean
    ↓
PermissionsContext (provides usePermissions hook)
    ↓
→ Route Guards (ProtectedRoute)
→ Sidebar (DashboardSidebar)
→ Component Actions (usePermissionCheck hook)
```

## 2. Permission Utilities (`src/utils/permissions.ts`)

Create a centralized permission utilities module with:

```typescript
// Core functions
canAccess(permission: string): boolean
hasAnyPermission(permissions: string[]): boolean  
hasAllPermissions(permissions: string[]): boolean
```

**Key behavior**:
- If `hasAllPermissions === true`, all functions return `true`
- Unknown permissions from backend are safely ignored (not in known permission map)
- Empty permissions array returns `false` for all checks

## 5. Route Guarding Strategy

### Updated ProtectedRoute Component

Modify existing `src/components/ProtectedRoute.tsx`:
- Accept optional `requiredPermission` prop
- Check permissions via context
- Redirect to fallback on unauthorized

### New PermissionRoute Component (`src/components/PermissionRoute.tsx`)

```typescript
interface PermissionRouteProps {
  children: ReactNode;
  requiredPermission?: string;
  fallbackPath?: string;
  showAccessDenied?: boolean;
}
```

**Behavior**:
1. If no `requiredPermission`, allow access
2. If `hasAllPermissions`, allow access
3. If user has required permission, allow access
4. Otherwise: redirect to `fallbackPath` or render "Access Denied" component

## 6. UI Integration

### Sidebar Navigation (`DashboardSidebar.tsx`)

Modify `getSidebarItems()` to accept permission filter:

```typescript
import { usePermissions } from '@/contexts/PermissionsContext';
import { SIDEBAR_PERMISSIONS } from '@/config/permissions';

function getSidebarItems(permissions: string[], hasAllPermissions: boolean): SidebarItemData[] {
  return SIDEBAR_PERMISSIONS
    .filter(item => hasAllPermissions || !item.requiredPermission || permissions.includes(item.requiredPermission))
    .map(item => ({ ...item, requiredPermission: undefined }));
}
```

### Component-Level Authorization

> **Simplified**: Component-level authorization is NOT in scope for this implementation. Navigation and route protection are the primary focus.

**Instead**: Navigation items will be filtered based on permissions. Individual action buttons (Add/Edit/Delete) can remain accessible - the assumption is that if a user can access a module, they have full CRUD capabilities within that module.

## 8. Edge Cases

| Scenario | Handling |
|----------|----------|
| Empty permissions array | `canAccess()` returns `false`, user gets minimal access |
| Unknown permissions | Only check against `KNOWN_PERMISSIONS` values; unknown = ignored |
| hasAllPermissions = true | Skip granular checks, return `true` for all |
| Permission check before context loads | Show loading state, don't render until ready |
| User navigates directly to unauthorized route | Route guard redirects to fallback |
